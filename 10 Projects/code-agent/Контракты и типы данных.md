# Контракты и типы данных

## Зачем нужен файл

Этот файл фиксирует общий протокол обмена данными между:
- `Orchestrator`
- `PlannerAgent`
- `WorkerAgent`
- `ConversationStore`
- `RAGService`

Главная цель — чтобы архитектура была согласована не только по ролям, но и по структурам данных.

## Основные сущности

### UserTask
Исходная задача пользователя.

Поля:
- `task_id: str`
- `conversation_id: str | None`
- `project_id: str | None`
- `text: str`
- `created_at: str`
- `meta: dict`

### PlanStep
Одна подзадача внутри плана.

Поля:
- `step_id: str`
- `title: str`
- `description: str`
- `priority: int`
- `status: str`  # pending / in_progress / done / failed / needs_replan
- `depends_on: list[str]`
- `tool_hint: str | None`

### PlanResult
Ответ planner'а.

Поля:
- `plan_id: str`
- `goal: str`
- `steps: list[PlanStep]`
- `reasoning_summary: str`
- `done: bool`
- `needs_worker: bool`

### WorkerResult
Ответ worker'а по конкретной подзадаче.

Поля:
- `step_id: str`
- `status: str`  # done / failed / needs_replan
- `output: str`
- `artifacts: list[dict]`
- `error: str | None`
- `reason: str | None`

### OrchestratorResult
Итог выполнения полной задачи.

Поля:
- `success: bool`
- `final_answer: str`
- `steps_completed: int`
- `replans: int`
- `messages_used: int`
- `artifacts: list[dict]`

### RetrievalChunk
Фрагмент контекста из RAG.

Поля:
- `chunk_id: str`
- `source_id: str`
- `project_id: str | None`
- `text: str`
- `score: float`
- `meta: dict`

### IngestResult
Результат индексации файла.

Поля:
- `source_id: str`
- `project_id: str | None`
- `chunks_count: int`
- `status: str`
- `message: str`

## Правила взаимодействия

### PlannerAgent
Вход:
- `UserTask`
- история беседы
- retrieval-контекст
- feedback от worker

Выход:
- `PlanResult`

### WorkerAgent
Вход:
- `PlanStep`
- retrieval-контекст
- история беседы
- tool context

Выход:
- `WorkerResult`

### Orchestrator
Вход:
- `UserTask`

Выход:
- `OrchestratorResult`

## Разрешённые статусы

Для шагов плана:
- `pending`
- `in_progress`
- `done`
- `failed`
- `needs_replan`

Для верхнеуровневых задач / заметок:
- `backlog`
- `planned`
- `active`
- `paused`
- `done`

## Важные принципы

- planner не выполняет tools напрямую
- worker не меняет глобальный план без возврата в orchestrator/planner
- retrieval-контекст не равен истории беседы
- project memory и global memory должны разделяться
- все структуры должны быть сериализуемыми в JSON
