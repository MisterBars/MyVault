# Архитектура multi-agent

## Идея

Проект развивается как локальный AI-ассистент с двумя основными ролями:
- PlannerAgent
- WorkerAgent

Planner получает исходную задачу пользователя, разбивает её на подзадачи и решает, в каком порядке они должны выполняться.

Worker получает конкретную подзадачу и пытается её выполнить.

Если worker понимает, что подзадача слишком сложная, плохо определена или требует дополнительной декомпозиции, он возвращает её обратно planner'у.

## Цепочка выполнения

```text
User
  ↓
Web UI / CLI
  ↓
Orchestrator
  ↓
PlannerAgent
  ↓
WorkerAgent
  ↓
Tools / RAG / Memory

При необходимости:
WorkerAgent → PlannerAgent → новая декомпозиция → продолжение выполнения
```

## Роли

### PlannerAgent
Отвечает за:
- анализ исходной задачи
- разбиение на подзадачи
- приоритизацию шагов
- перепланирование при возврате сложной подзадачи

### WorkerAgent
Отвечает за:
- выполнение конкретной подзадачи
- использование tools
- запрос retrieval-контекста
- возврат задачи planner'у, если задача оказалась слишком сложной

### Orchestrator
Отвечает за:
- координацию planner/worker
- хранение состояния текущего процесса
- контроль числа итераций
- защиту от бесконечной рекурсии
- сбор финального результата

## Ограничения первой версии

- сначала только текстовые ответы
- без сложных многошаговых инструментов
- без тяжёлой агентной иерархии
- одна цепочка planner → worker
- ограничение глубины перепланирования

## Принципы

- planner не выполняет действия напрямую
- worker не меняет глобальный план самостоятельно
- все внешние действия должны идти через tools
- retrieval и память беседы — разные слои
- архитектура растёт поэтапно
